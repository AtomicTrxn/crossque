# Research Topic #3 — Canvas + TalkBack/VoiceOver Accessibility

Status: Resolved
Implementation Status: ⬜ Pending — Deferred/Post-MVP; basic keyboard and touch interaction shipped, full CustomPainter semantics still needed
Owner: Codex

## Research Question

How should a Flutter `CustomPainter` crossword grid be made navigable and understandable for TalkBack, VoiceOver, keyboard users, and zoom users?

## Decision To Unblock

Can the app keep a custom canvas-rendered crossword grid without compromising screen reader accessibility, and what implementation pattern should be used before building the grid?

## Recommendation

Use a custom canvas for visual rendering, but make the grid semantics-first: every playable cell should have a matching semantic node generated by `CustomPainter.semanticsBuilder`, with explicit labels, values, focus/tap handlers, and traversal order. Do not rely on the visual canvas or a single wrapper `Semantics` node to describe the entire grid.

For MVP, implement per-cell semantics for all non-black cells in standard 15x15 and 21x21 puzzles. That is roughly 180-440 semantic nodes, which is acceptable enough to prototype and test early. If performance becomes poor on very large/variety grids, degrade to a row/active-word semantic model only after measuring.

## Key Findings

### Flutter Supports Semantics From `CustomPainter`

`CustomPainter` exposes `semanticsBuilder`, which returns `CustomPainterSemantics` rectangles describing parts of the painted canvas. Each semantic rectangle becomes a semantics node with `SemanticsProperties`.

Implementation implication: the grid painter should own both cell painting and cell semantics geometry so visual cell rectangles and accessible hit targets cannot drift apart.

### Cell Semantics Need Actions, Not Just Labels

Each playable cell should expose:

- `rect`: the exact cell rectangle in canvas coordinates.
- `label`: stable position and clue context, for example `Row 4, column 7. 12 across. Like a fox.`
- `value`: current entry state, for example `empty`, `A`, `rebus HEART`, `revealed A`, or `incorrect T`.
- `hint`: concise action guidance, for example `Double tap to select this cell.`
- `onTap`: select the cell and update puzzle focus.
- `onDidGainAccessibilityFocus`: update visual focus to match screen reader focus.
- `onFocus`: move input focus to the grid/keyboard handler if the platform requests input focus.
- `selected`: true for the active cell.
- `focused`: true for the one input-focused cell, false for other focusable cells; avoid deprecated `focusable`.
- `sortKey`: row-major `OrdinalSortKey` so swipe traversal is predictable.

Use the same selection handler for pointer taps and semantics taps. Flutter's semantics docs note that assistive technologies may call semantic actions or simulate physical taps, depending on platform and version.

### Traversal Order Should Be Explicit

Default platform traversal over a grid can be surprising, especially when cells are drawn on a canvas. Use `OrdinalSortKey(row * width + col)` for row-major traversal. Later, consider a solver setting for clue-order traversal, but start with row-major because it mirrors visual scanning and is easy to test.

### Announcements Should Mostly Be Implicit

Avoid frequent direct calls to `SemanticsService.sendAnnouncement` for every typed letter or cell movement. Flutter's current API says announcements are not supported everywhere, and Android discourages direct announcement events because they can interrupt TalkBack. Prefer updating the focused cell's semantics `value`, active clue `Semantics` label, and a polite `liveRegion` only for major state changes like word checked, puzzle completed, or source offline.

Use `SemanticsService.sendAnnouncement(View.of(context), ...)` only for exceptional messages that are not naturally represented by a changed semantic node, and guard it with platform support.

### Canvas Accessibility Requires Real Testing

Automated tests are useful but insufficient. The grid needs manual TalkBack and VoiceOver passes because accessibility focus, keyboard focus, and virtual keyboard behavior differ by platform.

Test with:

- Flutter accessibility guideline tests for labels, touch target size, and contrast.
- Android Accessibility Scanner.
- iOS Accessibility Inspector.
- Manual TalkBack: swipe through cells, double-tap select, type letters, check/reveal, zoom.
- Manual VoiceOver: same flows on iOS before Phase 2.

## Proposed Implementation Shape

```dart
class CrosswordGridPainter extends CustomPainter {
  CrosswordGridPainter({
    required this.grid,
    required this.focus,
    required this.direction,
    required this.onCellSelected,
    Listenable? repaint,
  }) : super(repaint: repaint);

  @override
  SemanticsBuilderCallback get semanticsBuilder => (Size size) {
        final metrics = GridMetrics.fromSize(size, grid.width, grid.height);
        final nodes = <CustomPainterSemantics>[];

        for (final cell in grid.playableCells) {
          final rect = metrics.cellRect(cell.row, cell.col);
          final isFocused = focus.row == cell.row && focus.col == cell.col;

          nodes.add(
            CustomPainterSemantics(
              rect: rect,
              properties: SemanticsProperties(
                label: cellSemanticsLabel(cell),
                value: cellSemanticsValue(cell),
                hint: 'Double tap to select this cell.',
                textDirection: TextDirection.ltr,
                selected: isFocused,
                focused: isFocused,
                sortKey: OrdinalSortKey(
                  (cell.row * grid.width + cell.col).toDouble(),
                  name: 'crossword-grid',
                ),
                onTap: () => onCellSelected(cell.row, cell.col),
                onDidGainAccessibilityFocus: () =>
                    onCellSelected(cell.row, cell.col),
                onFocus: () => onCellSelected(cell.row, cell.col),
              ),
            ),
          );
        }

        return nodes;
      };

  @override
  bool shouldRebuildSemantics(CrosswordGridPainter oldDelegate) {
    return oldDelegate.grid.semanticVersion != grid.semanticVersion ||
        oldDelegate.focus != focus ||
        oldDelegate.direction != direction;
  }
}
```

This is illustrative, not final code. The key design is that the painter receives enough immutable state to produce deterministic semantics from the same geometry it paints.

## Screen Reader Copy Model

| Element | Example |
|---------|---------|
| Empty active cell | `Row 4, column 7. 12 across. Like a fox. Empty.` |
| Filled cell | `Row 4, column 7. 12 across. Like a fox. Letter A.` |
| Checked incorrect cell | `Row 4, column 7. 12 across. Like a fox. Letter T. Marked incorrect.` |
| Revealed cell | `Row 4, column 7. 12 across. Like a fox. Revealed letter A.` |
| Rebus cell | `Row 4, column 7. 12 across. Like a fox. Rebus answer HEART.` |
| Black cell | Omit from semantics traversal. |
| Active clue bar | `12 across. Like a fox. 5 letters.` |
| Completion | Use dialog or live region: `Puzzle completed in 8 minutes 12 seconds.` |

Keep labels informative but not novel-length. The active clue bar can carry the full clue; cells can carry enough context to orient the user.

## Keyboard And Zoom Requirements

- Keep a real `FocusNode` on the grid container for hardware keyboard input.
- Arrow keys move cell focus; Space or Tab toggles direction; Backspace clears/moves according to solver settings.
- When accessibility focus lands on a cell, sync it to puzzle focus so keyboard entry targets the same cell.
- Pinch-to-zoom must update semantics rectangles after layout changes.
- Panning a zoomed grid should not leave offscreen semantics nodes with stale hit rectangles; test this explicitly. If needed, expose only visible cells while zoomed.

## Risks

| Risk | Mitigation |
|------|------------|
| Too many semantic nodes hurt performance on larger grids | Start with per-cell semantics, profile 15x15 and 21x21, and add visible-cell-only semantics for zoomed/panned large grids if needed. |
| Accessibility focus and input focus diverge | Use `onDidGainAccessibilityFocus`, `onFocus`, and the same `selectCell` action to synchronize state. |
| Direct announcements interrupt TalkBack | Prefer semantic value/live-region updates; reserve `sendAnnouncement` for exceptional events. |
| Custom keyboard is inaccessible | Keyboard keys must be normal widgets with their own semantics, labels, and actions, not part of the canvas. |
| Semantics drift from drawing geometry | Generate semantics rectangles from the same `GridMetrics` used by the painter. |

## Implementation Checklist

1. Build `GridMetrics` as a shared source of truth for paint, hit testing, and semantics rectangles.
2. Add `semanticsBuilder` to `CrosswordGridPainter` with per-playable-cell `CustomPainterSemantics`.
3. Exclude black cells from traversal.
4. Add `OrdinalSortKey` for deterministic row-major traversal.
5. Add semantic handlers for tap, input focus, and accessibility focus.
6. Add active clue bar as a separate `Semantics(liveRegion: true, ...)` only when clue changes need polite announcement.
7. Keep custom keyboard as normal Flutter widgets with button semantics.
8. Add widget tests for labels/actions and manual TalkBack/VoiceOver test scripts.

## Open Questions

- Should traversal default to row-major, clue order, or active-word-first? Recommendation: row-major for MVP, then test with screen reader users.
- Should black cells be announced as blockers? Recommendation: omit from traversal; the clue context gives better orientation.
- Should every letter entry trigger a live announcement? Recommendation: no; update cell value and let focus semantics do the work.
